# Report Review Checklist

Use this checklist before sharing a report more widely or relying on it for interpretation.

This checklist helps report consumers, report developers, department representatives, and workspace owners review whether a report is clear, safe, and fit for its intended purpose.

## 2. Purpose and Audience

* [ ] The report purpose is clear
* [ ] The intended audience is clear
* [ ] The report answers a clear business, operational, analytical, or learning question
* [ ] The report status is clear: sandbox, draft, prototype, UAT, department, or production
* [ ] Users can tell whether the report is official or experimental
* [ ] The report does not imply production readiness if it is only a sandbox or prototype artefact
* [ ] The report does not duplicate an existing approved report unnecessarily

## 3. Workspace Context

| Workspace Type           | Report Review Expectation                                                                                                             |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------- |
| Personal Workspace       | Suitable only for private drafts and individual exploration. Reports should not be treated as shared or official.                     |
| Sandbox Workspace        | Suitable for learning and experimentation. Reports are not official production outputs.                                               |
| Department Workspace     | Suitable for department-level working reports, prototypes, and approved department use cases. Owner and deputy owner should be known. |
| BIA Production Workspace | Suitable for BIA-managed production reports. Direct workspace access is restricted to BIA users.                                      |

Confirm:

* [ ] The report is in the correct workspace for its purpose
* [ ] Personal workspace reports are not being treated as shared department or production assets
* [ ] Sandbox reports are not being treated as official reports
* [ ] Department workspace reports have a clear workspace owner and deputy owner
* [ ] BIA Production Workspace access remains restricted to BIA users, where applicable
* [ ] Non-BIA users consume approved production outputs through approved report or app sharing channels, where applicable

## 4. Data Source and Refresh

* [ ] The data source is identified
* [ ] The report owner or contact is identified
* [ ] The refresh date or data currency is visible or documented
* [ ] Users can tell whether the data is current enough for the intended purpose
* [ ] Refresh ownership is clear, if refresh is required
* [ ] Known data limitations are documented
* [ ] Users can tell whether the report may contain stale data

## 5. Measures and Definitions

* [ ] Key measures are clearly named
* [ ] Key measures are defined
* [ ] KPI definitions are documented where needed
* [ ] Ambiguous terms are explained
* [ ] Users can understand what each metric means
* [ ] Measures are not duplicated or conflicting with known definitions
* [ ] Measures have been validated where appropriate
* [ ] Caveats are documented where definitions may be misunderstood

## 6. Layout and Usability

* [ ] Page titles are clear
* [ ] Visual titles are clear
* [ ] Visuals support the main question of the page
* [ ] The page is not overloaded with unnecessary visuals
* [ ] Slicers and filters are easy to understand
* [ ] Important filters are visible
* [ ] Navigation is intuitive
* [ ] Labels, legends, and axes are readable
* [ ] Users can identify what they are expected to look at first

## 7. Filters and Interactions

* [ ] Filters and slicers behave as expected
* [ ] Cross-filtering or cross-highlighting is intentional
* [ ] Users can tell when filters are active
* [ ] Default filter states are appropriate
* [ ] There are no confusing or misleading interactions
* [ ] Important context is not hidden by default filters
* [ ] Filtered views are not likely to be mistaken for the full picture

## 8. Interpretation and Caveats

* [ ] Important caveats are visible
* [ ] Users are warned where interpretation requires caution
* [ ] The report avoids overclaiming
* [ ] The report does not imply causality where only descriptive analysis is shown
* [ ] Limitations are documented
* [ ] Follow-up questions or context are provided where useful
* [ ] Users can distinguish between observation, interpretation, and conclusion

## 9. Access and Sharing

* [ ] The intended audience is authorised to view the report
* [ ] Sensitivity label is appropriate
* [ ] Sharing permissions are appropriate
* [ ] Export expectations are clear
* [ ] Screenshot or onward-sharing risks have been considered
* [ ] Row-Level Security has been considered where needed
* [ ] External collaborator access has been reviewed, if applicable
* [ ] Workspace access is not treated as permission to reuse, export, redistribute, or forward data
* [ ] If this is a BIA production report, non-BIA users access it through approved report or app sharing channels, not direct BIA Production Workspace membership

## 10. Validation

* [ ] Key values have been checked against source data or expected values
* [ ] Totals, averages, counts, and percentages behave as expected
* [ ] Filters do not produce unexpected results
* [ ] Report pages have been reviewed by the relevant owner or subject matter expert
* [ ] Known issues are documented
* [ ] The report is not shared as official if validation is incomplete
* [ ] Any unresolved validation issues are clearly documented

## 12. Final Checklist

* [ ] Report purpose is clear
* [ ] Intended audience is clear
* [ ] Workspace type and report status are clear
* [ ] Data source and refresh status are understood
* [ ] Key measures are defined
* [ ] Visuals are clear and usable
* [ ] Filters and interactions are intentional
* [ ] Caveats are visible
* [ ] Access and sensitivity have been reviewed
* [ ] Output has been validated
* [ ] Department workspace owner and deputy owner are identified, where applicable
* [ ] BIA Production Workspace access remains restricted to BIA users, where applicable
* [ ] Approved sharing channel is identified, where applicable
* [ ] Review decision is documented
