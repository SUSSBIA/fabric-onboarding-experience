# External Collaborator Checklist

Use this checklist before granting access to an external collaborator.

External collaborators may require additional review because they are not University staff and may need Microsoft Entra B2B guest access, licensing review, time-bound access, and stronger data sensitivity checks.

| Reviewer                      |          |

| Intended end date             |          |

## 2. Purpose of Access

* [ ] The purpose of access is clearly documented
* [ ] The collaborator’s role is clear
* [ ] The work is linked to an approved project, use case, engagement, or learning activity
* [ ] The expected output is clear
* [ ] The access requested is necessary for the stated purpose
* [ ] The access is not broader than needed
* [ ] The access duration is appropriate for the purpose

## 3. Workspace or Item Access

| Field               | Response                                                                           |
| ------------------- | ---------------------------------------------------------------------------------- |
| Workspace requested |                                                                                    |
| Workspace type      | Personal / Sandbox / Department / BIA Production / Not applicable                  |
| Item access needed  | Report / Semantic Model / Lakehouse / Notebook / Pipeline / Dataflow / App / Other |

| Requested role      | Viewer / Contributor / Member / Admin / Unsure                                     |
| Recommended role    | Viewer / Contributor / Member / Admin / Reject / Escalate                          |
| Access duration     | Temporary / Ongoing / Unsure                                                       |

## 4. Workspace Boundary Review

| Workspace Type           | External Collaborator Expectation                                                                         |
| ------------------------ | --------------------------------------------------------------------------------------------------------- |
| Personal Workspace       | Not suitable for external collaboration on University work                                                |
| Sandbox Workspace        | May be considered for approved learning, testing, or collaboration using safe data                        |
| Department Workspace     | May be considered only where there is an approved department use case, clear ownership, and access review |
| BIA Production Workspace | Direct workspace access should not be granted to external collaborators                                   |

Confirm:

* [ ] The requested workspace or item is appropriate for the stated purpose
* [ ] The access request does not use a personal workspace for shared work
* [ ] The access request does not use sandbox for official or production work
* [ ] The access request does not expose department assets beyond the approved purpose
* [ ] The request does not grant direct BIA Production Workspace access to the external collaborator
* [ ] If an approved production output must be shared, it will be shared through an approved report or app sharing channel instead of direct workspace membership

## 5. Department Workspace Ownership Review

Complete this section if access involves a department workspace.

* [ ] The department workspace owner is identified
* [ ] The deputy workspace owner is identified
* [ ] The owner or deputy has reviewed the external access request
* [ ] The owner or deputy understands the purpose and duration of access
* [ ] The owner or deputy understands who is responsible for access review and removal
* [ ] The request should not proceed if department workspace ownership is unclear

## 6. Microsoft Entra B2B Guest Access

* [x] Microsoft Entra B2B guest account requirement has been reviewed
* [ ] Guest account creation process is understood
* [ ] Guest identity is linked to the correct external email
* [ ] Access will be governed through the approved University identity process
* [ ] Guest access removal plan is identified
* [ ] Owner for guest access review or removal is identified

## 7. Licensing Review

* [ ] Power BI Pro or Premium Per User requirement has been reviewed
* [ ] It is clear whether the collaborator can use their own organisation’s license
* [ ] It is clear whether the University needs to assign a license
* [ ] Viewing-only scenario has been reviewed separately from editing or publishing
* [ ] Licensing requirement is confirmed before access is granted
* [ ] BIA or relevant administrator review is requested where needed

## 8. Data Sensitivity Review

* [ ] Data sensitivity has been assessed
* [ ] Sensitivity label is known or checked
* [ ] The collaborator is authorised to access the data
* [ ] Confidential or restricted data exposure has been reviewed
* [ ] Export and screenshot risks have been considered
* [ ] Sharing beyond the intended collaborator is not allowed unless approved
* [ ] The collaborator understands handling expectations
* [ ] The collaborator understands that access does not automatically allow reuse, export, redistribution, or onward sharing

## 9. Access Scope and Least Privilege

* [ ] Access is limited to the minimum required workspace, item, report, app, or dataset
* [ ] Viewer access is used where editing is not required
* [ ] Contributor or Member access is justified, if requested
* [ ] Admin access is avoided unless there is a specific approved reason
* [ ] Build permission is only granted if needed
* [ ] Access is time-bound where possible
* [ ] Review date is recorded
* [ ] Access can be removed when the purpose ends

## 10. Security and Governance

* [ ] External sharing policy has been considered
* [ ] Tenant-level settings are not bypassed
* [ ] Workspace role and item permission interaction is understood
* [ ] Row-Level Security is reviewed, if applicable
* [ ] The collaborator cannot access unintended content
* [ ] Any contractual, procurement, vendor-related, or data-sharing constraints are considered
* [ ] BIA review is completed where required
* [ ] Direct BIA Production Workspace access is not granted to the external collaborator

## 13. Final Checklist

* [ ] External collaborator is identified
* [ ] Purpose of access is documented
* [ ] Workspace, item, report, or app access is identified
* [ ] Access pattern is appropriate
* [ ] Microsoft Entra B2B guest account requirement is reviewed
* [ ] Licensing requirement is reviewed
* [ ] Minimum required access role is identified
* [ ] Data sensitivity is assessed
* [ ] External sharing risk is assessed
* [ ] Time-bound access is considered
